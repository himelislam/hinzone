# Claude Code - Backend Rules

## Purpose

These rules define the backend development standards for the **Stock Investment, Trading & MLM Platform**.

Every backend feature generated by Claude Code must follow these rules to ensure the system remains:

- Scalable
- Secure
- Modular
- Maintainable
- Testable
- Production Ready

This project uses:

- Node.js
- Express.js
- TypeScript
- MongoDB
- Mongoose

---

# 1. Backend Architecture

Always follow a feature-based modular architecture.

Example

```
src/

├── modules/
│   ├── auth/
│   ├── user/
│   ├── wallet/
│   ├── deposit/
│   ├── withdrawal/
│   ├── stock/
│   ├── trading/
│   ├── mlm/
│   ├── notification/
│   └── settings/
│
├── shared/
├── middleware/
├── config/
├── routes/
└── app.ts
```

Modules should remain independent whenever possible.

---

# 2. Clean Architecture

Every feature should follow this flow.

```
Route

↓

Middleware

↓

Controller

↓

Service

↓

Database

↓

Response
```

Controllers orchestrate requests.

Services contain business logic.

Models handle persistence.

---

# 3. Controller Rules

Controllers must only:

- Receive requests
- Validate request payloads
- Call services
- Return responses

Controllers must **never**:

- Query MongoDB directly
- Calculate commissions
- Modify wallet balances
- Execute business logic
- Access configuration values directly

Controllers should remain thin (typically under 150 lines).

---

# 4. Service Rules

All business logic belongs inside services.

Examples

- WalletService
- DepositService
- WithdrawalService
- TradingService
- PortfolioService
- MLMService
- SettingsService
- NotificationService

Services may call other services but should avoid circular dependencies.

---

# 5. Model Rules

Mongoose models should contain:

- Schema definitions
- Indexes
- Virtual fields
- Lightweight middleware (only when appropriate)

Do **not** place business logic in models.

---

# 6. Settings Service

The platform is fully configuration-driven.

Every configurable value must come from the Settings collection.

Never hardcode:

- Deposit limits
- Withdrawal limits
- Commission percentages
- Exchange rates
- Trading flags
- Maintenance mode
- Rank requirements
- Deposit packages
- Notification settings
- Security policies

Correct

```ts
const tradingSettings = await settingsService.getTradingSettings();
```

Incorrect

```ts
const TRADING_ENABLED = true;
```

---

# 7. Settings Cache

Application startup

```
MongoDB

↓

Settings Collection

↓

Memory Cache

↓

Application Ready
```

Whenever settings change

```
Update MongoDB

↓

Refresh Cache

↓

Immediate Effect
```

Never restart the server after changing settings.

---

# 8. Database Rules

Collections

- Users
- Wallets
- Transactions
- Deposits
- Withdrawals
- Stocks
- Portfolios
- Notifications
- MLM
- Settings
- AuditLogs

Use:

- Indexes
- References
- Timestamps
- Soft delete where appropriate

---

# 9. MongoDB Transactions

Every financial operation must use MongoDB transactions.

Examples

- Deposit Approval
- Withdrawal Approval
- Stock Purchase
- Stock Sale
- Wallet Adjustment
- MLM Bonus Distribution

Never allow partial writes.

---

# 10. Wallet Rules

Wallet balances are immutable except through WalletService.

Allowed operations

- Deposit
- Withdrawal
- Trading
- MLM Bonus
- Admin Adjustment

Every change creates:

- Transaction Record
- Audit Log (if administrative)

Never update balances directly with raw queries.

---

# 11. Authentication

Use

- JWT Access Token
- Refresh Token
- bcrypt Password Hashing

Never store:

- Plaintext passwords
- Refresh tokens in plaintext (if persisted)

Use secure cookie storage when applicable.

---

# 12. Authorization

RBAC roles

```
USER

ADMIN

SUPER_ADMIN
```

Protect every administrative route.

Permissions should be enforced through middleware.

---

# 13. Validation

Validate

- Body
- Query
- Params
- Headers
- Uploaded files

Recommended

```
Zod
```

Validation occurs before controllers execute.

---

# 14. API Response Format

Success

```json
{
  "success": true,
  "message": "Success",
  "data": {}
}
```

Failure

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": []
}
```

Never return inconsistent response formats.

---

# 15. Error Handling

Use custom error classes.

Examples

- ValidationError
- AuthenticationError
- AuthorizationError
- NotFoundError
- ConflictError
- BusinessRuleError

All exceptions should be processed by the Global Error Handler.

---

# 16. Logging

Use Winston.

Log

- Startup
- Shutdown
- API Errors
- Authentication Events
- Financial Events
- Unexpected Exceptions

Never log:

- Passwords
- JWTs
- Secrets
- Sensitive personal information

---

# 17. Middleware

Use reusable middleware.

Examples

- Authentication
- Authorization
- Validation
- Rate Limiting
- Error Handling
- Request Logging

Avoid duplicate middleware implementations.

---

# 18. Routes

Use RESTful naming.

Examples

```
GET

POST

PUT

PATCH

DELETE
```

Base path

```
/api/v1/
```

Group routes by feature.

---

# 19. Dependency Rules

Avoid circular dependencies.

If multiple services depend on each other, extract shared functionality into a dedicated service or utility.

---

# 20. Async Code

Always use

```ts
async / await;
```

Avoid nested promise chains.

Wrap controllers using a reusable async handler.

---

# 21. Repository Pattern

If repository complexity increases, introduce repositories.

```
Controller

↓

Service

↓

Repository

↓

MongoDB
```

For smaller modules, services may communicate directly with Mongoose models.

---

# 22. File Uploads

Supported uploads

- Avatars
- Deposit screenshots
- Homepage banners

Store files in

```
Cloudinary
```

Do not store uploads on the application server in production.

---

# 23. Security

Always use

- Helmet
- CORS
- Rate Limiting
- Input Sanitization
- HTTPS
- Secure Headers

Never trust:

- Request body
- Client-side calculations
- User role sent from frontend

---

# 24. Performance

Use

- Pagination
- Filtering
- Projection
- Lean queries where appropriate
- Indexes
- Settings cache

Avoid unnecessary database queries.

---

# 25. Background Jobs

Design services to support future background processing.

Future candidates

- Notifications
- Email
- Auto Sell
- Scheduled Rank Updates
- Cleanup Jobs

Potential stack

- Redis
- BullMQ

---

# 26. Notifications

NotificationService should be the single entry point for creating notifications.

Support

- User notifications
- Broadcast notifications
- Future email
- Future SMS
- Future push notifications

Notification preferences must be loaded from the Settings Service.

---

# 27. Audit Logs

Administrative actions should generate audit logs.

Examples

- Deposit approval
- Withdrawal approval
- Wallet adjustment
- Stock creation
- Settings update
- User suspension

Audit logs must be immutable.

---

# 28. Testing

Every module should include

- Unit tests
- Integration tests

Critical financial workflows require transaction and rollback testing.

---

# 29. Environment Variables

Never hardcode

- JWT secrets
- Database URLs
- Cloudinary credentials
- API keys
- SMTP credentials

Use environment variables only.

Environment files must never be committed.

---

# 30. Future Scalability

Write backend code that can support:

- Redis
- WebSockets
- Queue Workers
- Event-Driven Architecture
- Microservices
- GraphQL
- Multi-Tenant Deployments

Avoid tightly coupling modules.

---

# 31. Documentation

Every backend module should contain:

- README
- Route documentation
- Validation schemas
- Type definitions
- Tests

Keep documentation synchronized with implementation.

---

# 32. Non-Negotiable Rules

The following rules must always be followed:

- Controllers must remain thin.
- All business logic belongs in services.
- Wallet balances may only change through WalletService.
- Every financial operation must use MongoDB transactions.
- No configurable business rule may be hardcoded.
- All configurable values must be loaded through the Settings Service.
- Every request must be authenticated, authorized, and validated before executing business logic.
- Every API must return the standardized response format.
- Unexpected errors must be handled by the Global Error Handler.
- Administrative actions must generate Audit Logs.
- Sensitive information must never be exposed or logged.
- All code must be strongly typed with TypeScript.
- Backend code should always prioritize security, maintainability, and scalability over shortcuts.
