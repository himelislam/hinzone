# Claude Code - Coding Rules

## Purpose

These rules define the mandatory coding standards for the **Stock Investment, Trading & MLM Platform**.

Every file, function, component, API, and database operation generated by Claude Code **must follow these rules**.

The primary goals are:

- Maintainability
- Readability
- Scalability
- Performance
- Security
- Consistency

---

# 1. General Principles

Always follow:

- SOLID Principles
- DRY (Don't Repeat Yourself)
- KISS (Keep It Simple)
- YAGNI (You Aren't Gonna Need It)
- Clean Code
- Clean Architecture
- Separation of Concerns

Code should optimize for readability over cleverness.

---

# 2. TypeScript Rules

Always use:

```ts
strict: true;
```

Avoid:

```ts
any;
```

Prefer:

```ts
unknown;
```

or properly typed interfaces.

Always define explicit types for:

- Function parameters
- Return values
- API responses
- Database models
- Context values
- React props

---

# 3. File Size Limits

Maximum recommended sizes:

| File Type       |   Maximum |
| --------------- | --------: |
| React Component | 250 Lines |
| Service         | 300 Lines |
| Controller      | 150 Lines |
| Utility         | 200 Lines |
| Hook            | 200 Lines |

If a file exceeds these limits, refactor it into smaller modules.

---

# 4. Function Rules

Functions should:

- Have a single responsibility.
- Be easy to read.
- Return early when possible.
- Avoid deep nesting.

Avoid:

```ts
if (...) {
    if (...) {
        if (...) {
```

Prefer:

```ts
if (!condition) return;
```

---

# 5. Naming Conventions

## Variables

Use:

```ts
walletBalance;
userProfile;
depositAmount;
```

Never use:

```ts
x;
y;
temp;
abc;
```

unless in trivial loops.

---

## Functions

Use verbs.

Good

```ts
createDeposit();

approveWithdrawal();

calculateProfit();

getUserPortfolio();
```

Bad

```ts
deposit();

wallet();

profit();
```

---

## Boolean Variables

Prefix with:

```
is
has
can
should
```

Examples

```ts
isAdmin;

hasBalance;

canWithdraw;

shouldRefresh;
```

---

## Constants

Use:

```ts
MAX_UPLOAD_SIZE;
DEFAULT_PAGE_SIZE;
SUPPORTED_IMAGE_TYPES;
```

Never hardcode values that belong in the Settings Service.

---

# 6. Folder Rules

Follow the documented folder structure.

Never create:

```
misc/

temp/

helpers2/

newFolder/
```

Every file must belong to an existing feature or shared module.

---

# 7. React Rules

Use:

- Functional Components
- Hooks
- TypeScript
- Composition

Never use:

- Class Components
- Inline business logic
- Large JSX blocks

Extract reusable UI components.

---

# 8. React Component Structure

Recommended order:

```ts
Imports;

Types;

Constants;

Component;

Hooks;

Handlers;

Effects;

Return;
```

Keep components focused on presentation.

---

# 9. State Management

Use:

- React Query for server state.
- Context API for lightweight global state.
- Local state for component-specific UI.

Avoid storing server data in Context when React Query is appropriate.

---

# 10. API Rules

Always use:

```
/api/v1/
```

Never hardcode URLs.

Use environment variables.

Example

```ts
VITE_API_URL;
```

---

# 11. Controllers

Controllers should only:

- Validate input
- Call services
- Return responses

Controllers must **not**:

- Query MongoDB directly
- Perform calculations
- Apply business rules

---

# 12. Services

Services contain all business logic.

Examples

```
WalletService

DepositService

TradingService

MLMService

SettingsService
```

Services may call other services but should avoid circular dependencies.

---

# 13. Database Access

Database operations should be isolated.

Always:

- Use Mongoose models
- Handle errors
- Validate inputs
- Use transactions for financial operations

Never perform business calculations inside model definitions.

---

# 14. Settings Rules

All configurable values must come from the **Settings Service**.

Never hardcode:

- Deposit limits
- Withdrawal limits
- Exchange rates
- Commission percentages
- Rank requirements
- Trading flags
- Maintenance mode

Incorrect

```ts
const MIN_DEPOSIT = 3000;
```

Correct

```ts
const settings = await settingsService.getDepositSettings();
const minimumDeposit = settings.minimumDeposit;
```

---

# 15. Error Handling

Use custom error classes.

Example

```ts
ValidationError;

AuthenticationError;

AuthorizationError;

BusinessRuleError;

NotFoundError;
```

Throw meaningful errors and let the global error handler generate responses.

---

# 16. Logging

Use Winston for structured logging.

Log:

- Errors
- Authentication events
- Financial operations
- Startup and shutdown

Never log:

- Passwords
- JWTs
- API secrets
- Sensitive user data

---

# 17. Financial Operations

Every financial operation must use MongoDB transactions.

Examples

- Deposit approval
- Withdrawal approval
- Wallet adjustment
- Stock purchase
- Stock sale
- MLM commission distribution

Partial updates are not allowed.

---

# 18. Validation

Validate:

- Request body
- Query parameters
- URL parameters
- Uploaded files

Frontend validation improves UX but backend validation is mandatory.

---

# 19. Security

Always:

- Hash passwords with bcrypt
- Validate JWTs
- Sanitize input
- Enforce RBAC
- Apply rate limiting
- Use Helmet

Never trust client-provided values for financial calculations.

---

# 20. API Response Format

Success

```json
{
  "success": true,
  "message": "Operation completed successfully.",
  "data": {}
}
```

Error

```json
{
  "success": false,
  "message": "Validation failed.",
  "errors": []
}
```

Keep responses consistent across all endpoints.

---

# 21. Async/Await

Use:

```ts
async / await;
```

Avoid chaining `.then()` and `.catch()` in application code.

Wrap async controllers with a reusable async handler.

---

# 22. Imports

Order imports as follows:

1. Node.js modules
2. Third-party packages
3. Internal aliases
4. Relative imports

Remove unused imports automatically.

---

# 23. Comments

Write self-explanatory code.

Use comments only when necessary to explain:

- Complex algorithms
- Non-obvious business rules
- Performance optimizations

Avoid redundant comments.

---

# 24. Reusability

Before creating:

- Component
- Hook
- Utility
- Service
- Validation Schema

Check whether one already exists.

Prefer extension over duplication.

---

# 25. Performance

Avoid:

- Unnecessary renders
- N+1 database queries
- Duplicate API requests
- Blocking operations

Use:

- Pagination
- Projection
- Lean queries where appropriate
- React Query caching
- Memoization only when justified

---

# 26. Testing

Every new feature should include:

- Unit tests
- Integration tests
- Validation tests

Critical user flows should have end-to-end tests.

---

# 27. Git Rules

Write meaningful commit messages.

Examples

```
feat(wallet): implement transaction history

fix(trading): prevent duplicate stock purchases

refactor(settings): simplify cache refresh logic

test(deposit): add approval integration tests
```

Avoid generic messages such as:

```
update

fix

changes
```

---

# 28. Code Review Checklist

Before completing any feature, verify:

- No TypeScript errors
- No ESLint errors
- No duplicated logic
- Business rules use the Settings Service
- Validation is complete
- Authorization is enforced
- Error handling is implemented
- Tests pass
- Documentation is updated

---

# 29. Future-Proofing

Write code that can support future additions such as:

- Redis
- BullMQ
- WebSockets
- Mobile Apps
- Microservices
- Multi-Tenant Architecture
- Event-Driven Workflows

Avoid tightly coupling modules.

---

# 30. Non-Negotiable Rules

The following rules must **always** be followed:

- Never hardcode configurable business values.
- Never put business logic in controllers or React components.
- Never bypass the Settings Service.
- Never update wallet balances directly.
- Never expose sensitive information in API responses.
- Never commit secrets or environment files.
- Always use MongoDB transactions for financial operations.
- Always validate and sanitize user input.
- Always follow the documented folder structure.
- Always write strongly typed TypeScript.
- Always prefer reusable, modular solutions over shortcuts.
- If implementation details conflict with the project documentation, **the project documentation takes precedence**.
